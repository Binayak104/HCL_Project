# Error Log: 22-02-2026 - Infrastructure Refactoring

This document summarizes the technical challenges, root causes, and resolutions encountered during the migration to a 3-layer architecture and manual CI/CD workflows.

---

## 1. Missing Layer 1 Outputs
- **Error**: `Error: Unsupported attribute... data.terraform_remote_state.layer1.outputs.vpc_id`
- **Root Cause**: The root `infrastructure/layer1/main.tf` was calling the module but not explicitly exporting the `vpc_id` to its own output. Downstream layers cannot "see" module internal outputs.
- **Resolution**: Added `output "vpc_id" { value = module.layer1.vpc_id }` to the root Layer 1 configuration.

---

## 2. Hardcoded Availability Zone Mismatch
- **Error**: `Error: no matching Subnet found` in the Vector Layer.
- **Root Cause**: The infrastructure was searching specifically for a default subnet in `us-east-1b`. The user's VPC only had a subnet in `us-east-1a`.
- **Resolution**:
    - Updated Layer 1 to provision a **second public subnet** in `us-east-1b` automatically.
    - Updated the Vector Layer to dynamically find all available subnets in the VPC rather than relying on hardcoded discovery.

---

## 3. Undeclared Resource (Partial Application)
- **Error**: `Error: Reference to undeclared resource "aws_subnet.public_b"`
- **Root Cause**: A code update partially applied, adding the `aws_route_table_association` for the second subnet but failing to write the `aws_subnet` resource block itself.
- **Resolution**: Manually verified and restored the missing `aws_subnet.public_b` resource block in the Layer 1 module.

---

## 4. Missing Remote State Dependency
- **Error**: `Error: A data resource "terraform_remote_state" "layer1" has not been declared in module.layer2`
- **Root Cause**: During the transition to the 3-layer architecture, the `layer1` remote state (which provides ECR URLs) was accidentally removed or renamed when adding the `vector` state dependency.
- **Resolution**: Restored the `data "terraform_remote_state" "layer1"` block in the Layer 2 module so it can access both Networking/ECR and Database state.

---

## 5. RDS Connection Timeout (Security Group)
- **Error**: `Error initializing database: connection timed out` in Lambda logs.
- **Root Cause**: The RDS Security Group was restricted to VPC CIDRs only. Because the Lambda backend is currently running as a standalone service (not yet inside the VPC), it was blocked.
- **Resolution**: Updated the Vector Layer security group to allow ingress on port 5432 from `0.0.0.0/0` (public), utilizing the RDS instance's "Publicly Accessible" flag.

---

## 6. Build-Time vs Runtime Variable Conflict
- **Error**: `getaddrinfo ENOTFOUND placeholder-url`
- **Root Cause**: The frontend was using `NEXT_PUBLIC_API_URL`. Next.js "bakes" these into the build. Since the build happened in GitHub Actions with a placeholder, the running app was trying to call `placeholder-url` instead of the real API Gateway.
- **Resolution**:
    - Refactored the frontend proxy to use a **runtime-only** variable: `API_BASE_URL`.
    - Updated the Next.js `middleware.ts` to read this variable only during request processing.
    - Updated Terraform to inject the real API Gateway endpoint into the App Runner configuration.

---

## 7. App Runner Concurrent Update Lock
- **Error**: `InvalidStateException: Service cannot be updated in the current state: OPERATION_IN_PROGRESS`
- **Root Cause**: The GitHub Action was deploying code to App Runner at the same time Terraform was trying to update the service's environment variables. AWS blocks concurrent updates.
- **Resolution**: Monitored the service status via CLI. Once the code deployment reached `RUNNING`, re-triggered the Terraform apply to finalize the environment configuration.

---

**Status**: All issues resolved. System is functional.
